Detail View — Option A: Grid with Diff Highlighting

The current multi-column grid, but with a key improvement: source values that match the result are dimmed. Values that differ from the result stay bright. This makes differences jump out instantly — you scan for color, not content. The grid becomes a diff view.

  • Result column values: always bold/bright (this is your current state)
  • Source values matching result: dim (nothing to do here)
  • Source values differing from result: normal/yellow (this would change if applied)
  • Missing values: dim ·

Detail View — Option B: One Source at a Time

Instead of showing all sources as cramped columns, show result vs. one source side by side. Use h/l to cycle through sources. This gives each value generous horizontal space and reduces the comparison to a simple left-right diff.

Navigation becomes 1D (just rows) instead of 2D (rows and columns). enter on a row applies that field. ⇧enter applies all fields from the current source. h/l switches which source you're comparing against.

Filename Source: a Design Question

Currently, filename (guessit extraction) is just another source you cycle through with h/l. But it's fundamentally different from TMDB results — it's the starting point, not an alternative match. Three ways to handle this:

Option 1: Filename is source [1/N] (current)

Simple, consistent. To "reset" a field, cycle to the filename source and apply. Downside: filename feels like it's on the same level as TMDB matches, which is conceptually wrong.

Option 2: Filename not a source — reset with D

Only TMDB results are sources. d clears to empty, D (shift-d) resets to the filename-extracted value. The filename extraction is the "ground truth" you can always fall back to.

├─ Detail ──────────────────────────────────────────────────────────────────────────────────┤
│ Inception.2010.1080p.BluRay.x264-GROUP.mkv │
│ → Inception (2010)/Inception (2010).mkv │
│ ────────────────────────────────────────────────────────────────────────────────────────── │
│ result TMDB #1 (92%) [1/2] │
│ ────────────────────────────────────────────────────────────────────────────────────────── │
│ title Inception Inception │
│ year 2010 2010 │
│ media_type movie movie │
│ codec x264 · ← from filename (x264) │
│ source BluRay · ← from filename (BluRay) │
│ │
└──────────────────────────────────────────────────────────────────────────────────────────────────┘
enter apply e edit d clear D reset to filename h/l sources esc back ? help
Where did "x264" come from? In this model, the result starts with filename-extracted values, then TMDB values are layered on top (auto-accept). Fields like codec and source that TMDB doesn't provide keep their filename values. A dim annotation could show the origin when it's not obvious. D on any field resets it to the filename extraction. d clears to empty.

Option 3: Filename as "base layer" with origin markers

The result always shows where each value came from. Small dim markers after the value indicate origin. This makes the data lineage visible without needing a separate source.

Origin markers: Each result value has a small dim label showing where it came from — tmdb, file, or edit (for manually edited values). This answers "why does my result say this?" at a glance. D resets a field back to the filename value. d clears to empty. Could also be shown as icons instead of text to save space.

Comparison: Grid vs. One-at-a-Time

Option A (Grid with diff highlighting) keeps the multi-column layout. You see all sources at once. Good for quickly identifying which source is best. Risk: with 3+ sources, columns get tight and the screen gets dense. The diff highlighting helps, but you're still parsing a 2D grid.

Option B (One source at a time) shows just result vs. one source. Navigation is 1D (rows only). Each value gets generous space. The comparison is focused and calm. Switching sources with h/l is fast. Risk: you can't see all sources simultaneously, so comparing across sources requires memorizing or cycling. But the confidence % in the header tells you which source is likely best, and the color system (dim=match, yellow=differ, green=fills gap) gives instant feedback.

My recommendation: Option B. The one-at-a-time layout is more intuitive, easier to parse, and works perfectly at any terminal width. The grid is powerful but cognitively expensive — most users will auto-accept the best source anyway. The detail view is for the edge cases where you need to cherry-pick, and focused comparison makes that easier.

The diff color system works for both options — it's an independent improvement.
